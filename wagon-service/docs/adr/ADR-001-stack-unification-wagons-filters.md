# ADR-001: Унификация стека для GET /api/v1/wagons/filters

## Статус

**Принято** — 2026-02-28

## Контекст

Ранний артефакт разработчика описывал реализацию эндпоинта `GET /api/v1/wagons/filters` в NestJS (TypeScript), при этом в строке таблицы стека была оговорка «Python (FastAPI) или NestJS — по проекту». Это создавало неопределённость: в каком сервисе живёт эндпоинт, какой стек канонический.

Исходные факты:

- `wagon-service` — существующий сервис, написанный на **Python (FastAPI)** с SQLAlchemy (async) и Alembic-миграциями.
- NestJS-сервиса в проекте **нет** — он упоминался как гипотетическая альтернатива.
- Эндпоинт `/api/v1/wagons/filters` — read-only аналитический запрос (4× DISTINCT), без stateful-логики, без сложных преобразований.
- Требование SRS: ≤ 3 с при 500 вагонах. Текущий стек выполняет его тривиально.

## Рассмотренные варианты

| Вариант | Описание | За | Против |
|---------|----------|----|--------|
| **A (принято)** | Реализовать в существующем FastAPI `wagon-service` | Нет нового сервиса; общий gateway; единый деплой; переиспользование ORM-моделей, миграций, auth | Нет |
| B | Вынести в отдельный NestJS-микросервис | Возможность использовать TypeScript-экосистему | Новый сервис ради одного эндпоинта — overengineering; дублирование авторизации и моделей; два деплоя |
| C | Двухстековый подход (FastAPI + NestJS параллельно) | Позволяет тестировать NestJS-стек инкрементально | Нет бизнес-обоснования; операционная сложность удваивается |

## Решение

**Вариант A: реализовать GET /api/v1/wagons/filters в существующем FastAPI `wagon-service`.**

Двухстековый подход и введение NestJS откладываются до момента, когда появится реальный сценарий, где TypeScript-экосистема даёт неоспоримое преимущество. Для четырёх DISTINCT-запросов этого преимущества нет.

## Последствия

- Все файлы эндпоинта живут в `wagon-service` (Python/FastAPI):
  - Роутер: `app/api/v1/wagons.py`
  - Сервис: `app/services/wagon_service.py`
  - Репозиторий: `app/repositories/wagon_repo.py`
  - Схемы: `app/schemas/wagon.py` (`WagonFiltersResponse`, `ClientOption`)
  - Миграция индексов: `alembic/versions/0005_add_filter_distinct_indexes.py`
- Если в будущем появится NestJS-сервис — его можно подключить через тот же API-gateway без изменения контракта фронтенда.
- Артефакт разработчика обновлён: строка «Python (FastAPI) или NestJS» заменена на однозначное «Python/FastAPI (`wagon-service`)».

## Технологический стек (итог)

| Слой | Технология | Обоснование |
|------|------------|-------------|
| БД | PostgreSQL | Уже используется в проекте |
| Индексы | B-Tree (partial, WHERE deleted_at IS NULL) | Достаточны для DISTINCT по varchar-колонкам; partial исключает soft-deleted записи |
| BE | **Python/FastAPI**, `wagon-service` | ADR-001: единый сервис, общий gateway и деплой |
| ORM | SQLAlchemy 2.0 (async) | Уже используется во всех запросах `wagon-service` |
| Миграции | Alembic | Единый механизм миграций проекта |
| Авторизация | JWT + `require_any_role` dependency | Переиспользуется из существующей auth-инфраструктуры |
| Кэш | Не вводим на первом этапе | DISTINCT по 500 строкам ≈ 5–50 мс; Redis добавит сложность без выгоды |

## Почему не Redis на первом этапе

SRS требует ≤ 3 с при 500 вагонах. При partial-индексах каждый DISTINCT-запрос выполняется за 5–15 мс, четыре запроса — за 10–20 мс суммарно. Запас до лимита — 150×. Если датасет вырастет до десятков тысяч строк — добавим materialized view или Redis отдельным ADR.

## API-контракт

### GET /api/v1/wagons/filters

**Request:**

```
GET /api/v1/wagons/filters
Authorization: Bearer <token>
```

Параметры запроса отсутствуют — возвращаем все distinct-значения из текущих данных.

**Response 200:**

```json
{
  "destination_railways": ["string"],
  "clients": [{"name": "string"}],
  "suppliers": ["string"],
  "destination_stations": ["string"]
}
```

**Response 401:** Unauthorized — невалидный или отсутствующий токен.
**Response 403:** Forbidden — роль без доступа к рабочему месту логиста.

## Открытый вопрос: идентификатор клиента

Текущая реализация использует `Request.client_name` (строковый). После решения AC-UC-1.2-2 возможен переход на `client_id` — это потребует добавления поля `id` в `ClientOption` без breaking change (поле опционально).
